# WebSocket STOMP, Redis와  RabbitMQ 
WebSocket	웹 기반, 앱 기반 실시간 양방향 통신 (기본)
STOMP (on WebSocket)	구독/전송 규칙을 쉽게 적용하고 싶을 때 (ex: 채팅방, 구독 기능)
MQTT	IoT 장비처럼 초경량 메시지가 필요한 경우
AMQP (RabbitMQ)	신뢰성 높은 비동기 메시지 통신이 필요할 때 (ex: 결제, 주문 처리)
Redis Pub/Sub	빠른 서버 간 실시간 동기화가 필요할 때 (ex: 실시간 채팅방 동기화  서버가 여러대일 때 )

✅ 웹(브라우저) 기반 채팅 서버를 만들 때 기본 흐름
기본 전송 계층은 WebSocket을 사용한다.
WebSocket만 사용해서 채팅을 구현해도 되지만   
구독 관리, 세션 관리, 방 분리 등을 개발자가 직접 세세하게(for문 등) 개발하기 어려우니 STOMP 사용


✅ Redis나 RabbitMQ는 언제 쓰나?
서버가 1대일 때는 필요 없다.
서버가 여러 대로 늘어날 때
(ex: 사용자 트래픽 분산, 서버 장애 대비 등)
이때 서버 간 메시지 공유를 위해 메시지 브로커가 필요하고,
대표적으로 Redis Pub/Sub 또는 RabbitMQ를 사용한다.

✅ Redis와 RabbitMQ 선택 기준 (간단)

| 상황 | 선택 |
| --- | --- |
| 속도 최우선, 휘발성 허용 | Redis Pub/Sub |
| 신뢰성 최우선, 메시지 저장/복구 필요 | RabbitMQ |
| 경우에 따라 둘 다 혼용 | ex: 채팅은 Redis, 결제는 RabbitMQ |

🎯 요약
WebSocket(전송) + STOMP(애플리케이션) + Redis/RabbitMQ(서버 간 메시지 공유)



# 채팅에서의 구독 
채팅 개발에서 말하는 "구독(subscription)"은 아주 간단히 말하면,
"내가 특정 채널이나 대상을 계속 주시하고 있다가, 거기에 **새로운 메시지나 이벤트가 발생하면 알림을 받는 것"


🔵 채팅 개발에서 "구독"이 필요한 이유
사용자는 특정 채팅방이나 **특정 주제(topic)**에 들어가 있죠.
그런데 서버는 "누가 어느 채팅방에 들어와 있는지" 알아야 새 메시지가 생길 때마다 그 사람한테 실시간으로 알려줄 수 있어야 합니다.
이걸 "구독한다"라고 표현해요.

🔵 구독은 어떤 기술로 구현하는가?
보통 WebSocket + STOMP 프로토콜 기반에서는,
클라이언트가 서버에 SUBSCRIBE 명령을 보내면서
"나는 /topic/chatroom/123을 구독할게" 이렇게 요청합니다.

클라이언트 
SUBSCRIBE
destination:/topic/chatroom/123

서버
/topic/chatroom/123 이라는 주제(topic)에 새 메시지가 발생할 때마다
이 채팅방을 구독한 클라이언트들에게 PUBLISH(전송) 합니다.

🔵 
[채팅방1]
   ↳ 사용자 A (구독함)
   ↳ 사용자 B (구독함)
[채팅방2]
   ↳ 사용자 C (구독함)
채팅방1에 새 메시지 발생 → 사용자 A, B한테만 즉시 보내줌
채팅방2에 새 메시지 발생 → 사용자 C한테만 즉시 보내줌
