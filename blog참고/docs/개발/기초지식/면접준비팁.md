# 개발 면접 준비 메모

## 핵심 한 줄

> **"나 진짜 잘해!"가 아니라, "나는 이 조직에 맞는 사람이야!"** 가 되어야 한다.

기술 자랑 대회가 아니다. 회사가 원하는 문제를 풀 수 있는 사람인지를 보여주는 자리다.

---

## 1. 이력서(지원서) — 질문을 "유도"하도록 쓰기

이력서는 자랑 목록이 아니라, **면접관이 물어보게 만드는 미끼**다.

- 각 항목을 **`어떤 기술 → 어떻게 적용 → 어떤 문제를 해결`** 구조로 적는다.
- 내가 강조하고 싶은(=잘 답할 수 있는) 주제로 질문이 흐르도록 유도한다.

> ⚠️ **주의:** 이력서에 쓴 기술은 **무엇이든 깊이 답할 수 있어야** 한다.
> 실무에서 적용할 때는 깊이 공부하지 않았더라도, **면접 전에는 반드시 파고들어** 둔다.

**예시 (한 줄 → 파생 질문)**

| 이력서에 쓴 것 | 면접에서 나올 법한 질문 |
| --- | --- |
| "RabbitMQ로 주문 처리를 비동기화했다" | 왜 메시지 큐? Kafka는 왜 아니었나? 메시지 유실은 어떻게 막았나? |
| "JPA N+1을 fetch join으로 해결했다" | N+1이 왜 생기나? fetch join 말고 다른 방법은? |

---

## 2. 아키텍처 / 시스템 설명 준비하기

면접관은 **코드 한 줄**이 아니라, 내가 **말한 구조**를 기준으로 파고든다.

![면접 질문은 어디서 나오는가](개발면접%20잡다한것_image/question-flow.svg)

### 미리 정리해 둘 것들

- 서버 몇 개인지, **앞단 서버 / 필터**가 있는지
- **관리자 서버**를 따로 구성했는지
- **운영 / 개발 / 로컬** 환경은 어떻게 나뉘는지
- **메일 서버**가 따로 있는지
- **RabbitMQ 같은 메시지 큐**를 쓰는지, **버전**은 무엇인지
- **DB는 어떤 식으로 나뉘어**(읽기/쓰기 분리, 샤딩 등) 있는지

### 포인트

- 이 설명을 듣고 나서 **후속 질문이 파생**된다.
- **실제 했던 작업 기반**으로 물어보지, 단순 코드 레벨은 잘 안 물어본다.
- 그래서 **말로만이 아니라 그림으로** 설명할 수 있게 준비하면 좋다 (흐름도, 서버 아키텍처, 배포 환경 구성 등).

---

## 3. 경험 — 무엇을 어필할까

### ① 혼자 끝까지 해본 경험
- 처음부터 끝까지 스스로 굴려본 경험 (사이드 프로젝트 등)

### ② 학습 가능성
- 백엔드 Spring 기반이지만, **최신 스택**(Java 21, Kotlin 등)도 써봄
- 새로운 기술을 스스로 학습해 적용해본 경험

### ③ 협업 · 의사소통
- **코드 리뷰 / GitHub**를 어떻게 활용했는지
- **요구사항 문서가 자세하지 않을 때** 어떻게 소통해서 풀었는지

---

## 4. 경험은 "STAR" 틀로 말하기

"어떤 문제를 어떻게 해결했는지"를 두서없이 말하면 전달이 안 된다.
**상황 → 과제 → 행동 → 결과** 네 단계로 정리하면 설득력이 생긴다.

![경험은 STAR 틀로 말하기](개발면접%20잡다한것_image/star.svg)

- **여러 프로젝트를 얕게 "찍먹"한 경험은 큰 도움이 안 된다.**
- 하나라도 **어떤 문제를 어떻게 해결했는지**를 STAR로 깊이 있게 녹여내는 게 낫다.
- 결과(R)에는 가능하면 **수치**를 넣는다 (응답 시간, 처리량, 에러율 등).

---

## 5. "회사에 맞춘" 자기소개

- 회사 **분야에 맞는** 자기소개를 준비한다 — 기술 스택과 경험을 그 회사 맥락에 맞춰 재구성.
- **무엇이든 그림을 그리면서** 설명한다 (흐름도, 서버 아키텍처, 배포 환경 구성 등).

---

## 한눈에 요약

1. 목표: **"잘하는 사람"이 아니라 "이 조직에 맞는 사람"** 임을 보여주기
2. 이력서: 질문을 유도하도록 쓰고, 쓴 건 다 답할 수 있게 공부
3. 아키텍처: 실제 구조를 그림까지 준비 (코드 레벨보다 구조/의사결정을 물음)
4. 경험: 혼자 해본 것 + 학습 가능성 + 협업, STAR로 깊이 있게
5. 회사 맞춤: 그 회사 분야에 맞춰, 그림 그리며 설명
